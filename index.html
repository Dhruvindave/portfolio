03 / Practice

How I ship code

As CTO at Swikar Technology, I owned the release process end to end — this is the branching and deployment discipline I built and enforced.

Branching & release strategy

git-flow, self-designed

Two permanent branches — master for production, staging for internal validation — with every developer working in their own isolated branch, named so nobody steps on anybody else's work.

Branch master→ dev_name-branch_name
Review merge request→ CTO reviews & resolves conflicts→ merge to staging
Validate build Docker image→ deploy to staging servers→ validate
Release merge to master→ GitHub release, versioned & tagged
Each feature branch is cut fresh from master, so every developer always builds on the latest production-ready code — even while working across multiple branches in parallel.
If staging validation fails, the change is kicked back to the dev branch and the cycle repeats. Nothing reaches master unvalidated.
